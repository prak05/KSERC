       1. What the Project Is About

       KSERC ARA (Autonomous Regulatory Agent) is a backend system

       The regulatory process it automates:
       - Electricity distribution companies (licensees like
       Infopark, Technopark, KDHPCL) submit petitions each financial
        year.

       revenues against the previously approved Annual Revenue
       Requirement (ARR).
       - KSERC needs to verify mathematical accuracy, identify
       deviations, and assess compliance.

       regulatory orders, extract financial tables, calculate
       deviations, and produce compliance reports.

       3. Features

       1. PDF Ingestion & Table Extraction -- Upload KSERC
       regulatory order PDFs; the system extracts financial tables
       using pdfplumber, parses licensee name (regex pattern M/s
       [Company Name]), and financial year (regex pattern year
       YYYY-YY).
       2. Financial Analysis -- Extracts rows of financial data
       (expense categories), compares ARR-approved amounts vs.
       actual ("trued up") amounts, and calculates deviations per
       line item.
       3. Compliance Checks -- Three automated checks:
         - Mathematical Accuracy: Verifies that totals match the sum
        of individual rows (zero-error math).
         - Significant Deviation Analysis: Flags items with >10%
       deviation; severity is HIGH (>20%) or MEDIUM (10-20%).
         - Surplus/Deficit Assessment: Checks if net surplus/deficit
        exceeds 5% of total ARR.
       4. Executive Summary Generation -- Produces a structured
       summary with financial overview, deviation breakdown
       (overspent/underspent/on-budget counts), largest deviation
       identification, and compliance status.
       5. DataFrame Export -- Converts results to Pandas DataFrames
       for potential Excel/CSV export or further statistical
       analysis.
       6. REST API with Auto-Documentation -- Swagger UI at /docs
       and ReDoc at /redoc (in debug mode).
       7. Custom Colored Logging -- ANSI color-coded console output
       with configurable log levels, optional file logging, a
       function-call decorator for tracing, and a standardized
       exception logger.
       8. Fallback/Sample Data -- If no financial tables are
       extracted from a PDF, the system falls back to hardcoded
       sample data (Power Purchase Cost, Employee Expenses, Repair &
        Maintenance) so the API always returns a valid response.

       Detailed Logic Flow

       POST /analyze-order/ (src/main.py ->
       src/services/pdf_ingestion.py):
       1. Validates file is .pdf
       2. Checks file size against MAX_UPLOAD_SIZE (50MB default)
       3. Reads file bytes asynchronously
       4. Calls process_regulatory_order(file_bytes) which:
         - Opens PDF via pdfplumber from a BytesIO stream
         - Extracts full text from all pages
         - Runs regex to find licensee name (M/s [Name] pattern)
         - Runs regex to find financial year (year YYYY-YY or FY
       YYYY-YY pattern)
         - Extracts all tables from all pages using pdfplumber's
       table detection
         - Parses the first table with valid rows into FinancialRow
       objects (skipping header row, requiring 3+ columns)
         - Cleans currency strings (removes commas, rupee symbols,
       extracts numeric values)
         - Calculates deviation per row: trued_up_value -
       arr_approved
         - Computes totals and net surplus/deficit: total_arr -
       total_actual
         - Falls back to sample data if no tables are extracted
         - Returns a TruingUpResponse Pydantic model

       POST /compliance-check/ (src/main.py ->
       src/services/analyzer.py):
       1. Runs the same PDF processing as /analyze-order/
       2. Then calls perform_compliance_checks(analysis_result)
       which runs:
         - Check 1 - Mathematical Accuracy: Verifies
       sum(row.arr_approved) == total_arr_approved and
       sum(row.trued_up_value) == total_trued_up (within 0.01
       tolerance)
         - Check 2 - Significant Deviations: Calls
       identify_significant_deviations() with a 10% threshold; items
        >20% get severity "HIGH", 10-20% get "MEDIUM"
         - Check 3 - Surplus/Deficit Assessment: Calculates
       |net_surplus_deficit| / total_arr * 100 and flags if it
       exceeds 5%
       3. Then calls generate_analysis_summary() which produces:
         - Financial overview (totals, net surplus/deficit)
         - Deviation breakdown (overspent/underspent/on-budget
       counts)
         - Key insights (largest deviation item, compliance status,
       warning count)
         - Compliance summary (checks passed/failed/warnings)
       4. Returns a combined comprehensive_report dict with
       basic_analysis, compliance_report, and executive_summary

       Data Models (src/models/schemas.py):

       - FinancialRow: particulars (str), arr_approved (float),
       trued_up_value (float), deviation (float) -- includes a
       Pydantic validator ensuring deviation == trued_up_value -
       arr_approved within 0.01 tolerance
       - TruingUpResponse: licensee_name, financial_year,
       financial_summary (List[FinancialRow]), net_surplus_deficit,
       total_arr_approved, total_trued_up, analysis_timestamp
       (auto-set to datetime.now()), compliance_status
       - ErrorResponse: error, message, details (optional dict)
       - HealthCheckResponse: status, system, timestamp, version

       Configuration (src/config.py):

       A Settings class reads from environment variables (via
       python-dotenv) with sensible defaults: host 0.0.0.0, port
       8000, debug mode on, 50MB upload limit, PDF DPI 300, table
       detection tolerance 3px.

       - Per-endpoint try/except with HTTPException for 400/500
       errors
       - Global exception handler returns JSON error responses
       - Debug mode controls whether error details are exposed to

       - Tests all 4 endpoints sequentially
